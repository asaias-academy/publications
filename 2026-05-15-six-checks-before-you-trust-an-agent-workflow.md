# Six Checks Before You Trust an Agent Workflow

Status: approved public publication v1  
Release date: 2026-05-15  
Format: practical publication

## Purpose

This publication offers a compact institutional checklist for deciding whether an agent workflow is ready for serious use. It is written for managers, founders and operators who need a practical frame before scale.

## The six checks

### 1. Can you describe the workflow boundary clearly?

If the unit of work is vague, trust will be vague as well. Define:

- what task is delegated;
- what completion means;
- what lies outside the workflow.

### 2. Can you see where context came from?

If a system uses retrieved or injected context, an institution should know:

- what sources were used;
- how current they are;
- what remains uncertain.

### 3. Can you see what tools were involved?

Trust weakens quickly when tool access is implicit. A serious workflow should make visible:

- what systems it can call;
- what actions it can take;
- what it cannot do.

### 4. Can you inspect the path of execution?

A workflow becomes more governable when the path of execution can be reviewed:

- steps taken;
- handoffs made;
- retries or corrections;
- notable uncertainty points.

### 5. Do evaluation and review exist before scale?

Capability claims should not stand alone. Institutions need some combination of:

- eval routines;
- review criteria;
- repeatable checks;
- correction logging.

### 6. Is final human approval real?

Final approval matters only if it is meaningful. A human approval layer becomes stronger when the human reviewer can actually inspect the relevant path, evidence and uncertainty before acceptance.
